# Resourceful To-Do

Спокойный планировщик дня с упором на реальный запас сил, а не на абстрактную продуктивность.

Приложение помогает:
- планировать день по энергии;
- держать под рукой ресурсные дела и радости;
- мягко переносить лишнее в `На потом`;
- сохранять завершённые дела в `Сделано`;
- вести историю самочувствия;
- работать с аккаунтом и серверным сохранением состояния.

## Что сейчас внутри

Проект состоит из двух частей:
- frontend приложения в папке `js/`, который собирается в `js/app.bundle.js`;
- backend на Node.js + Express, который раздаёт сайт, хранит пользователей и состояние приложения.

Серверное хранение уже переведено на SQLite:
- база находится в `data/app.db`;
- старые `data/users.json`, `data/state.json` и `data/states/*.json` нужны только как legacy-источник для мягкой миграции.

## Быстрый локальный запуск

1. Откройте PowerShell в папке проекта `C:\ToDo`.
2. Установите зависимости:

```powershell
npm install
```

3. Создайте файл `.env` из примера:

```powershell
Copy-Item .env.example .env
```

4. Соберите браузерный файл:

```powershell
npm run build
```

5. Запустите сервер:

```powershell
npm start
```

6. Откройте в браузере:

```text
http://localhost:3000
```

## Что такое `.env`

`.env` — это файл с настройками сервера.

Он нужен для того, чтобы:
- не править настройки прямо в коде;
- легко переключаться между локальным запуском и будущим запуском в интернете;
- хранить параметры сервера отдельно от исходников.

Файл `.env.example` — это просто образец.

Файл `.env` — это ваш настоящий рабочий файл.

По умолчанию в нём можно оставить такие значения:

```env
PORT=3000
NODE_ENV=development
DATABASE_PATH=data/app.db
TRUST_PROXY=false
SESSION_COOKIE_NAME=rtodo_sid
SESSION_COOKIE_SAME_SITE=Lax
SESSION_COOKIE_SECURE=false
SESSION_TTL_HOURS=168
```

## Чем отличаются `development` и `production`

### `development`

Это режим для работы на своём компьютере.

В нём:
- сервер запускается удобно для локальной разработки;
- cookie могут работать без HTTPS;
- в консоли больше полезной технической информации;
- ошибки легче разбирать.

Для локального запуска обычно достаточно:

```env
NODE_ENV=development
SESSION_COOKIE_SECURE=false
TRUST_PROXY=false
```

### `production`

Это режим для будущего размещения в интернете.

В нём:
- сервер работает строже и осторожнее;
- cookie должны быть готовы к HTTPS;
- сервер может стоять за proxy или хостинг-платформой;
- пользователю не показываются лишние технические детали ошибок.

Для будущего боевого запуска обычно понадобятся такие значения:

```env
NODE_ENV=production
SESSION_COOKIE_SECURE=true
TRUST_PROXY=true
```

Важно: `production` имеет смысл только там, где приложение действительно запускается через внешний сервер и HTTPS.

## Основные серверные настройки

- `PORT` — порт, на котором запускается сервер.
- `NODE_ENV` — режим работы: `development` или `production`.
- `DATABASE_PATH` — путь к SQLite-базе.
- `TRUST_PROXY` — нужен, если приложение запускается за proxy или на хостинге.
- `SESSION_COOKIE_NAME` — имя cookie сессии.
- `SESSION_COOKIE_SAME_SITE` — политика cookie: обычно `Lax`.
- `SESSION_COOKIE_SECURE` — требует HTTPS, если `true`.
- `SESSION_TTL_HOURS` — срок жизни сессии в часах.

## Что уже готово для будущего размещения

Сейчас проект уже умеет:
- запускаться через локальный Express-сервер;
- хранить пользователей и состояние в SQLite;
- работать с регистрацией, входом и выходом;
- читать настройки из `.env`;
- различать `development` и `production` режимы;
- аккуратно готовиться к переносу на внешний хостинг.

## Что ещё не сделано

Пока проект всё ещё остаётся промежуточной версией перед настоящим деплоем:
- сессии пока живут только в памяти сервера;
- внешний домен и HTTPS ещё не подключены;
- проект ещё не развернут на хостинге;
- production-база пока всё ещё SQLite, а не отдельный сервер баз данных.

## Git и локальные данные

В репозиторий не должны попадать:
- `node_modules/`
- `data/`
- `.env`

Поэтому:
- `.env` хранится только у вас локально;
- `data/app.db` тоже остаётся только на вашей машине;
- в GitHub можно отправлять исходники, но не личные данные и не рабочую базу.
