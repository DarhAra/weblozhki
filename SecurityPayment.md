# Безопасная платёжная архитектура и защита данных

## Цель

Сделать добровольную поддержку проекта простой для пользователя и безопасной для приложения.

Главный принцип:
- приложение не должно самостоятельно обрабатывать или хранить платёжные данные
- приложение должно хранить только минимум информации, необходимой для работы
- личные данные пользователя должны рассматриваться как чувствительные

---

## Общая стратегия

### 1. Вынести оплату на сторону платёжного провайдера

Пользователь не вводит платёжные данные внутри приложения.

Правильная модель:
- в приложении есть кнопка поддержки
- после нажатия пользователь переходит на безопасную страницу оплаты платёжного провайдера
- после успешного платежа пользователь возвращается обратно в приложение

Это снижает риски и упрощает архитектуру.

---

### 2. Не доверять только возврату пользователя после оплаты

Возврат на экран приложения после оплаты сам по себе не должен считаться подтверждением успешного платежа.

Подтверждением считается только серверное уведомление от платёжного провайдера.

---

### 3. Подтверждать платежи только по webhook

Сервер принимает webhook от платёжного провайдера и проверяет его подлинность.

Только после успешной проверки:
- платёж помечается как успешный
- в профиле пользователя обновляется статус поддержки
- при необходимости создаётся запись в истории платежей

---

## Что приложение должно хранить

Приложение должно хранить только минимально необходимое.

Допустимо хранить:
- внутренний id пользователя
- id платежа у платёжного провайдера
- статус платежа
- тип платежа
- сумму
- дату платежа
- дату окончания поддержки, если она ограничена по времени

---

## Что приложение не должно хранить

Не хранить:
- номера карт
- CVC/CVV
- полные платёжные реквизиты
- любые платёжные поля, введённые пользователем на странице оплаты
- лишние персональные данные, не нужные для работы продукта

---

## Минимизация данных

Нужно придерживаться принципа минимизации.

Это означает:
- собирать только то, что действительно нужно
- не сохранять “на будущее” лишние поля
- не держать в базе чувствительную информацию без явной причины
- не сохранять содержимое личных записей в логах
- не отправлять личные тексты пользователя в сторонние сервисы без отдельного основания

Особенно бережно нужно относиться к:
- облаку мыслей
- истории самочувствия
- low energy и SOS-сценариям
- голосовым черновикам
- персональным заметкам и задачам

---

## Сессии и авторизация

### 1. Не хранить авторизационные данные в localStorage

`localStorage` не должен использоваться для:
- токенов
- session id
- признака авторизации
- любых секретов, дающих доступ к аккаунту

Если локальное хранилище используется как временный фолбэк, в нём должны оставаться только нечувствительные черновики и данные, не дающие доступ к аккаунту.

---

### 2. Использовать серверные сессии через cookie

Сессия пользователя должна работать через безопасные cookie.

Для cookie должны быть заданы защитные свойства:
- `Secure`
- `HttpOnly`
- `SameSite`

Также важно:
- ограничить срок жизни сессии
- обновлять session id после входа
- обновлять session id после чувствительных действий
- корректно завершать сессию при выходе

---

### 3. Работать только по HTTPS

Всё приложение должно работать только через защищённое соединение.

Нельзя:
- использовать HTTP для страницы входа
- смешивать защищённые и незащищённые ресурсы
- передавать сессионные cookie по незашифрованному каналу

---

## Пароли

Пароли должны храниться только в виде стойких хэшей.

Требования:
- не хранить пароли в открытом виде
- не использовать быстрые алгоритмы общего назначения
- использовать специально предназначенный алгоритм для хранения паролей
- параметры хэширования должны быть настроены осознанно, а не оставлены слабыми по умолчанию

---

## Webhook-безопасность

Webhook от платёжного провайдера должен обрабатываться осторожно.

Обязательные требования:
- проверять подпись webhook
- не принимать неподписанные запросы
- не доверять данным без верификации
- защищаться от повторной отправки одного и того же события
- обрабатывать повторные вызовы идемпотентно
- хранить только нужный результат обработки

---

## Безопасность логов

Логи не должны становиться утечкой чувствительных данных.

Не писать в логи:
- тексты из облака мыслей
- содержимое личных задач
- историю самочувствия
- голосовые черновики
- email + личный контент в одном сообщении
- данные webhook целиком, если в этом нет необходимости

В логах лучше хранить:
- технический тип события
- внутренний id пользователя
- внутренний статус операции
- код ошибки без чувствительного содержимого

---

## Безопасность клиентской части

Клиентская часть не должна считаться доверенной средой.

Поэтому:
- нельзя подтверждать платежи только на клиенте
- нельзя менять права пользователя только по ответу браузера
- нельзя рассчитывать на скрытие логики в интерфейсе как на защиту
- нельзя считать локальные данные доказательством авторизации или оплаты

Все критические решения должны приниматься сервером.

---

## Безопасность пользовательских данных

Приложение работает с психологически чувствительным контекстом, даже если это не медицинский продукт.

Поэтому важно:
- относиться к данным пользователя как к приватному пространству
- минимизировать копирование личных текстов
- не использовать личные данные в аналитике без необходимости
- давать пользователю ощущение контроля над его данными
- предусмотреть понятное удаление данных
- не показывать лишние данные в интерфейсе и уведомлениях

---

## Безопасная модель поддержки проекта

Для добровольной поддержки безопасная схема должна быть такой:

1. Пользователь нажимает кнопку поддержки в приложении
2. Приложение отправляет его на безопасную платёжную страницу провайдера
3. Пользователь оплачивает поддержку на стороне провайдера
4. Платёжный провайдер отправляет webhook на сервер
5. Сервер проверяет подпись webhook
6. Сервер обновляет статус поддержки пользователя
7. Пользователь возвращается в приложение и видит корректный результат

---

## Что важно не делать

Не стоит:
- делать собственную форму ввода карты
- сохранять платёжные данные у себя
- подтверждать платёж только через редирект
- хранить session id в localStorage
- смешивать чувствительные данные с техническими логами
- собирать больше данных, чем реально нужно

---

## Приоритеты внедрения

### Первый приоритет
- вынести оплату на hosted checkout платёжного провайдера
- принимать подтверждение только через webhook
- проверять подпись webhook

### Второй приоритет
- убрать auth-данные и session id из localStorage
- перевести авторизацию на безопасные cookie
- включить HTTPS везде

### Третий приоритет
- пересмотреть состав хранимых пользовательских данных
- сократить объём логов
- убедиться, что чувствительный контент не попадает в диагностику и аналитику

### Четвёртый приоритет
- усилить параметры хранения паролей
- улучшить завершение и обновление сессий
- подготовить понятное удаление пользовательских данных

---

## Итоговый принцип

Платёжная система должна быть простой не только для пользователя, но и для архитектуры.

Лучший подход:
- платёжные данные обрабатывает внешний провайдер
- приложение хранит только минимум нужных сведений
- все критические статусы подтверждаются сервером
- личные данные пользователя защищаются по принципу минимизации и бережного обращения

Это соответствует философии приложения:
спокойно, безопасно, без лишнего вторжения в личное пространство.
